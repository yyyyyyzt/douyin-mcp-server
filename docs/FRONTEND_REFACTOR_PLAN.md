# 自装助手 · 前端重构开发规划

> 目标：把当前偏工程工具的 WebUI，重构为完全不懂 AI 的用户也能在手机上顺手使用的自装助手。
> 后续 agent 开发前必须先读本文件，再读 `docs/DESIGN.md` 与 `PROGRESS.md`。
> 微信原生小程序前端应复用本文件的信息架构与文案原则，实现细节见
> [`WECHAT_MINIPROGRAM_PLAN.md`](WECHAT_MINIPROGRAM_PLAN.md)。
> **小程序 UI 布局与组件库优化**见 [`MINIPROGRAM_UI_OPTIMIZATION.md`](MINIPROGRAM_UI_OPTIMIZATION.md)。

---

## 1. 产品定位

- **项目名称**：自装助手。
- **目标用户**：第一次装修、主要用手机操作、不理解 API Key / LLM / ASR / Prompt / 结构化等 AI 概念的普通用户。
- **核心场景**：
  1. 刷到装修短视频，复制分享链接，快速存成自己的装修知识。
  2. 装修现场或与工长沟通时，打开手机查自己的知识卡片。
  3. 对报价单、合同、施工问题提问，答案必须说明是否来自个人知识库。
- **体验目标**：少解释、少配置、少输入；每屏只放一个主动作；所有技术状态翻译成人话。

---

## 2. 当前前端现状

- 入口文件：`web/templates/index.html`，原生 HTML + Tailwind CDN + Alpine.js，单页应用，无构建流程。
- 当前能力：知识收集、知识库、问答三 Tab；模型设置弹窗；PWA manifest + service worker。
- 主要问题：
  - 顶部导航和多列布局更像桌面工具，手机端单手操作不够顺。
  - “转写 / AI 整理 / 入库 / 模型设置”等概念对 AI 零基础用户不友好。
  - 知识收集页信息密度高，链接输入、整理预览、原始文案同时出现，首次使用成本高。
  - 配置状态过早暴露，用户看到“未配置密钥”容易困惑。
  - 聊天页可用，但缺少装修场景化快捷问题和答案可信度表达。

---

## 3. 信息架构

底部固定 3 个 Tab，所有手机端主路径都从这里进入：

| Tab | 建议文案 | 用户理解 | 主动作 |
|---|---|---|---|
| 收集 | `收集` | 把看到的装修经验存起来 | 粘贴链接 / 粘贴文字，点“保存到知识库” |
| 知识库 | `知识库` | 查看自己存过的装修经验 | 搜索、筛选、查看、编辑、删除 |
| 问答 | `问答` | 问自己的装修知识 | 输入问题、上传报价单、查看依据 |

原则：
- Tab 数量固定为 3 个，不新增“设置”Tab。
- 设置入口放在头像/状态胶囊/更多按钮中，默认隐藏高级项。
- Tab 文案保持短词，不使用“提取”“结构化”“检索”等技术词。

---

## 4. 全局 UI 原则

### 4.1 手机优先

- 主容器按 390px 宽手机优先设计，再向平板/桌面扩展。
- 底部固定导航高度建议 64px，加 `safe-area-inset-bottom`。
- 页面底部内容必须预留导航遮挡空间：`padding-bottom: calc(80px + env(safe-area-inset-bottom))`。
- 主要按钮固定在当前卡片或页面底部，拇指可触达。
- 点击目标不小于 44px，高频按钮不小于 48px。

### 4.2 极简表达

- 每屏只突出一个主按钮。
- 默认隐藏原始文案、模型选择、技术错误详情；需要时用“查看详情”展开。
- 用户文案使用结果导向：
  - “转写”改为“读取视频内容”。
  - “AI 整理”改为“整理成知识”。
  - “入库”改为“保存到知识库”。
  - “grounded=false”提示改为“这条回答没有找到你的知识依据”。
  - “API Key 未配置”改为“服务还没连接好，请先完成一次初始化”。

### 4.3 降低 AI 感

- 前台不出现 LLM、ASR、Prompt、模型、幻觉等词。
- 需要展示可信度时用“有依据 / 没找到依据”，不要说“防幻觉”。
- 模型设置作为高级设置保留，默认折叠；普通用户路径不依赖手动选模型。

### 4.4 视觉方向

- 品牌名：自装助手。
- 风格：干净、生活化、轻工具；避免过强“抖音红”占比。
- 推荐色彩：
  - 主色：暖橙或珊瑚红，用于主按钮和关键状态。
  - 背景：米白 / 浅灰。
  - 成功：绿色。
  - 警告：琥珀色。
- 卡片圆角与阴影保持克制，避免页面像后台管理系统。

## 5. 页面规划

### 5.1 收集 Tab

目标：用户打开后只知道一件事——把链接粘进去保存。

首屏结构：
1. 顶部问候与一句话说明：`把刷到的装修经验存成自己的知识。`
2. 大输入框：支持抖音分享链接、整段分享文案、纯文字。
3. 主按钮：`保存到知识库`。
4. 进度卡片：用 3 步人话状态展示：
   - `正在读取视频`
   - `正在整理重点`
   - `已保存`
5. 最近保存成功的卡片预览。

交互要求：
- 默认走“一键保存”：链接或文字提交后自动完成读取、整理、保存。
- 如果需要用户确认，放在成功后“查看并修改”，不要阻断主流程。
- 原始文案默认折叠为“查看原文”。
- 失败时给下一步建议，例如“链接可能过期了，试试重新复制分享链接”。

后端适配建议：
- 当前已有 `/api/video/extract`、`/api/cards/save` 等接口；前端可先串联调用。
- 后续可新增单个“收集”聚合 API，但不要在第一阶段强依赖后端改造。

验收标准：
- 手机宽度下，用户不滚动即可看到输入框和主按钮。
- 粘贴文字可以保存成知识卡片。
- 无 API Key 时提示人话，不出现 `API_KEY`、`ASR`、`LLM`。

### 5.2 知识库 Tab

目标：像查看备忘录一样查看装修知识。

结构：
1. 顶部搜索框：`搜防水、报价、验收...`
2. 横向阶段筛选 chips：全部 / 水电 / 防水 / 泥木 / 油漆 / 验收 / 其他。
3. 卡片列表：标题、摘要、来源、保存时间。
4. 卡片详情底部抽屉或独立详情状态：正文、步骤、来源链接、编辑/删除。

交互要求：
- 搜索框始终在顶部，不藏在设置里。
- 卡片点击进入详情，不在列表中堆太多按钮。
- 编辑与删除放在详情页或更多菜单，删除必须二次确认。
- 空状态给行动：`还没有知识，去收集第一个装修经验`。

验收标准：
- 100 张卡片以内滚动流畅。
- 搜索和阶段筛选在手机端可单手使用。
- 删除/编辑不会误触。

### 5.3 问答 Tab

目标：让用户放心地问，但知道答案有没有依据。

结构：
1. 空状态快捷问题：
   - `卫生间防水要注意什么？`
   - `帮我看报价单有没有漏项`
   - `水电验收要检查哪些地方？`
2. 聊天区。
3. 附件按钮：上传报价单/合同。
4. 输入框 + 发送按钮。

答案展示：
- 有引用：显示绿色小标 `来自你的知识库`，引用折叠为 `查看依据`。
- 无引用：显示琥珀提示 `这条回答没有找到你的知识依据，仅供参考`。
- 上传文件分析：区分 `来自报价单` 与 `来自知识库`。

验收标准：
- 首次打开问答页时，不是空白聊天框，而是可点的装修问题建议。
- 用户能一眼看出答案是否来自自己的知识库。
- 上传文件按钮在手机端易点，不挤压输入框。

## 6. 组件与代码组织建议

当前无前端构建流程，第一阶段仍保持 `web/templates/index.html` 单文件，避免引入 React/Vue 构建成本。

建议在单文件内先整理出清晰区块：

- `<!-- App Shell -->`：顶部品牌、全局状态、底部 Tab。
- `<!-- Collect Tab -->`：收集页。
- `<!-- Library Tab -->`：知识库页。
- `<!-- Ask Tab -->`：问答页。
- `<!-- Modals / Sheets -->`：设置、编辑、详情抽屉。
- `renoApp()` 内按 `collect / library / ask / settings` 分组字段和方法。

若后续页面继续膨胀，再评估拆为静态 JS/CSS 文件：

- `web/static/app.css`
- `web/static/app.js`

拆分时仍保持零构建，不能影响 Cloud Agent 和自托管启动方式。

## 7. 分阶段实施计划

### 阶段 1：命名与移动端 App Shell

- 全量改名为“自装助手”。
- 顶部品牌简化，移除顶部 segmented Tab。
- 新增底部固定 3 Tab：收集 / 知识库 / 问答。
- 处理 safe area 与底部内容遮挡。
- 保留所有现有功能，不做接口变更。

验收：
- `python -m pytest` 通过。
- 手机浏览器下底部 Tab 固定可用，三个页面可切换。
- PWA 名称显示为“自装助手”。

### 阶段 2：收集流程极简化

- 将“链接转写 + AI 整理 + 入库”包装为单一主流程。
- 收集页只保留一个大输入框和一个主按钮。
- 进度文案改成人话。
- 整理预览改为成功后的可选编辑。

验收：
- 粘贴纯文字可保存。
- 粘贴有效链接可走完整保存流程。
- 失败提示给用户可执行建议。

### 阶段 3：知识库移动端重排

- 增加搜索框和阶段 chips。
- 卡片详情改为底部抽屉或详情视图。
- 列表减少按钮密度，编辑/删除进入详情后操作。

验收：
- 列表在 390px 宽度下信息清晰。
- 搜索、筛选、查看、编辑、删除链路可用。

### 阶段 4：问答可信度表达

- 添加快捷问题。
- 优化引用展示为“查看依据”。
- 无依据提示改为用户语言。
- 上传附件状态更明确。

验收：
- 有依据答案、无依据答案、带附件答案三种状态都能清晰区分。
- 用户不需要理解“知识库检索”也能知道答案来源。

### 阶段 5：PWA 与细节打磨

- 校准 manifest 名称、主题色、图标。
- 优化 iOS Safari 键盘弹起、底部输入框遮挡、安全区。
- 增加 loading skeleton、toast、离线外壳提示。

验收：
- 可添加到手机主屏幕，名称为“自装助手”。
- iOS/Android 常见视口下无底部遮挡。

## 8. 后续 agent 开发规则

1. 每个阶段尽量独立提交，避免一次性重写整个 `index.html`。
2. 改 UI 必须做手机视口手动验证，并保留截图或录屏。
3. 不改变现有 API 契约，除非对应后端测试先覆盖。
4. 技术词只能出现在高级设置或错误详情中，不能出现在主流程。
5. 新增文案优先用短句，避免“智能”“大模型”“Prompt”等词。
6. 保持 PWA 可用：manifest、service worker、图标路径不能断。
7. 完成阶段后同步更新 `PROGRESS.md`。
